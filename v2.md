# ClaudeTrade V2 Production Design

V2 is production trading infrastructure, not a test harness. The system must preserve the core split:

Claude creates the investment hypothesis. Runtime code executes it safely. The database records wins, losses, misses, blocks, and dirty data. The brain only absorbs slowly verified patterns.

Safety Gate protects the account. Timing Gate chooses entry timing. Neither gate replaces Claude's alpha judgment.

Profit is the final goal, but V2 does not convert profit directly into belief. It decomposes weak returns into selection, entry, exit, sizing, data quality, and cost.

## Execution Identity

Every decision, event, execution, and position record must carry:

- `market`: `KR` or `US`
- `runtime_mode`: `live` or `paper`
- `decision_id`
- `execution_id` when an order attempt exists
- `position_id` when an actual filled position exists
- `prompt_version`
- `brain_snapshot_id`

`decision_id` is issued at `CLAUDE_TRADE_READY`. Entry failure, safety block, timing expiry, order reject, partial fill, and unknown order states are all learning records.

Paper data is research/shadow data. In V2, paper data is never automatically merged into the live brain, even if it is `CLEAN`.

## Initial KR/US Launch Policy

The first live launch enables both KR and US, but each market is capped to one live position and one daily entry.

```text
TARGET_SESSION_DATE = 2026-04-27
ENABLED_MARKETS = KR,US
KR_LIVE_START = enabled with 100000 KRW source size
US_LIVE_START = enabled with 100000 KRW source size
```

US sizing must not use a pre-guessed USD amount. V2 stores the US source size in KRW and uses the latest available USD/KRW rate at runtime to convert US prices into KRW before sizing and safety checks.

Start config:

```text
config/v2_start_config.json
```

Initial live start values:

```text
KR_FIXED_ORDER_KRW = 100000
US_FIXED_ORDER_KRW = 100000
KR_MIN_ORDER_KRW = 100000
US_MIN_ORDER_KRW = 100000
KR_MAX_POSITIONS = 1
US_MAX_POSITIONS = 1
DAILY_LOSS_LIMIT_PCT = -2.0
V2_MAX_DAILY_ENTRIES = 1
entry_mode = next_open
timing_adapter = momentum_timing
```

Sunday preparation is offline only. It must not place live orders, send broker order calls, or call Claude just to test plumbing. The allowed work is code/path verification, dry-run/replay of lifecycle records, start config validation, brain policy verification, and report generation.

Brain start policy:

```text
V2_BRAIN_POLICY = fresh_v2_reference_v1
```

V1 brain data is kept as reference/archive, but it is not injected into V2 live prompts for the first KR start. V2 live brain candidates still require `CLEAN` live data after forward measurement.

## Daily Flow

```text
SCOUT
-> JUDGE
-> DECISION_REGISTRY
-> SAFETY_GATE
-> TIMING_ADAPTER
-> ORDER_EXECUTOR
-> POSITION_MANAGER
-> FORWARD_MEASURED
-> QUALITY_MARKED
-> LEARNING
-> DAILY_REVIEW
```

SCOUT/JUDGE policy:

- Pre-session SCOUT/JUDGE: once per market before open
- Opening refresh: once 10-20 minutes after open, only when candidates changed materially
- Mid-session SCOUT: allowed every 60 minutes
- Claude JUDGE: maximum 2 calls per market per day
- Hold Review: separate from JUDGE and only reviews open positions

Mid-session JUDGE is allowed only when at least one condition is true:

- current `trade_ready` count is 0
- index intraday move exceeds `MIDSESSION_INDEX_MOVE_TRIGGER_PCT = 1.0`
- candidate set changed by at least `MIDSESSION_CANDIDATE_CHANGE_RATIO = 0.5`
- an existing candidate changed to stale or suspect market data
- operator explicitly requested it

Hold Review default:

```text
HOLD_REVIEW_INTERVAL_MINUTES = 60
```

Immediate Hold Review triggers:

- hard stop proximity
- trailing stop candidate
- 30 minutes before close
- broker sync mismatch
- operator request

## Lifecycle

```text
CLAUDE_TRADE_READY
-> SAFETY_PASSED / SAFETY_BLOCKED
-> WAIT_TIMING / TIMING_UNSUPPORTED / TIMING_EXPIRED
-> ORDER_SENT
-> ORDER_ACKED
-> PARTIAL_FILLED / FILLED
-> ORDER_REJECTED / ORDER_CANCELLED / ORDER_UNKNOWN
-> CLOSED
-> FORWARD_MEASURED
-> QUALITY_MARKED
```

`TIMING_EXPIRED`, `TIMING_UNSUPPORTED`, `SAFETY_BLOCKED`, and order failures are still `FORWARD_MEASURED` targets. They may have no `actual_trade_result`, but Claude selection quality must still be measured through `forward_from_decision` and `selection_alpha`.

## Safety Gate

Safety Gate is deterministic account protection. A new safety rule is accepted only when all are true:

- it prevents real account damage
- it is deterministic
- it is not an alpha judgment
- it has a recordable `reason_code`

Initial reason codes:

```text
INSUFFICIENT_CASH
ALREADY_HOLDING
PENDING_ORDER_EXISTS
MAX_POSITIONS
MAX_DAILY_ENTRIES
DAILY_LOSS_LIMIT
BROKER_UNTRUSTED
INVALID_PRICE
MARKET_CLOSED
MIN_ORDER_NOT_MET
STALE_MARKET_DATA
SAME_DAY_REENTRY_AFTER_STOP
ORDER_UNKNOWN_UNRESOLVED
```

Initial launch safety values:


## Timing Adapter

Timing Adapter only decides when to buy.

Initial implementation:

```text
momentum_timing
```

Momentum timing requires high breakout, volume increase, and short-term trend hold.

Unsupported timing nature is recorded as `TIMING_UNSUPPORTED`.

Initial TTL:

```text
MOMENTUM_TIMING_TTL_MINUTES = 45
PULLBACK_TIMING_TTL_MINUTES = 120
NEW_ENTRY_CUTOFF_MINUTES_BEFORE_CLOSE = 10
```

## Orders, Partial Fill, Unknown Orders

Partial fill policy:

```text
partial fill
-> issue position_id only for filled quantity
-> wait for remaining quantity until TTL
-> request cancellation after TTL
-> record ORDER_REMAINDER_CANCELLED when confirmed
-> record ORDER_UNKNOWN when cancellation/fill truth is unresolved
```

Initial TTL:

```text
KR_PARTIAL_FILL_TTL_SEC = 120
US_PARTIAL_FILL_TTL_SEC = 180
```

Performance is calculated only from actual filled quantity. Unfilled remainder is excluded from position PnL.

ORDER_UNKNOWN escalation:

- 1 unresolved case: pause new orders for that ticker and re-query broker truth
- 2 consecutive cases in same market: pause new entries for that market
- KR and US both untrusted, or balance mismatch: pause all new entries until `/resume`

## Position Manager

Close reason enum:

```text
CLOSED_HARD_STOP
CLOSED_TRAILING_STOP
CLOSED_TIME_STOP
CLOSED_CLAUDE_SELL
CLOSED_USER_MANUAL
CLOSED_SESSION_FORCE
CLOSED_MARKET_HALT
CLOSED_PANIC
CLOSED_BROKER_SYNC
```

Time Stop:

```text
elapsed since entry >= market time stop minutes
and MFE < TIME_STOP_MIN_MFE_PCT
and current return < TIME_STOP_MIN_PROGRESS_PCT
-> CLOSED_TIME_STOP
```

Units are percent-point returns versus entry price:

```text
KR_TIME_STOP_MINUTES = 90
US_TIME_STOP_MINUTES = 120
TIME_STOP_MIN_PROGRESS_PCT = 0.3  # +0.3%
TIME_STOP_MIN_MFE_PCT = 0.5       # +0.5%
```

`CLOSED_BROKER_SYNC` is an internal state repair event and is not counted as actual trade performance.

## Performance

Performance is split into:

- `forward_from_decision`: 1/3/5 day return from Claude decision time
- `actual_trade_result`: actual entry to actual close result
- `selection_alpha`: `forward_from_decision - benchmark_forward`
- `entry_delay`
- `MFE`
- `MAE`
- `exit_efficiency`
- `net_pnl_after_claude`

Benchmarks:

- KR KOSPI stocks: KOSPI
- KR KOSDAQ stocks: KOSDAQ
- US default: SPY
- US Nasdaq/tech/growth: QQQ auxiliary benchmark

MDD displays:

- `since_inception_mdd`
- `rolling_20d_mdd`
- `rolling_60d_mdd`

## Data Quality

Grades:

```text
CLEAN
SUSPECT
DIRTY
LEGACY_UNKNOWN
```

Rule engine runs:

- after event append
- during daily review
- after forward measurement

Initial rules:

- `FILLED` without `ORDER_SENT`: `DIRTY`
- `CLOSED` without `FILLED` or `PARTIAL_FILLED`: `DIRTY`
- unresolved `ORDER_UNKNOWN`: `SUSPECT`
- `FORWARD_PENDING_DATA`: learning blocked
- `CLOSED_BROKER_SYNC`: `SUSPECT` or `DIRTY`

Brain candidate generation default:

```text
runtime_mode = live
data_quality = CLEAN
forward measurement complete
```

## Prompt And Brain

`brain_snapshot_id` is created immediately before Claude JUDGE.

Snapshot contains:

- `prompt_version`
- brain pattern list
- brain hash
- `market`
- `session_date`
- `runtime_mode`
- `created_at`

Pattern grades:

```text
sample < 10: observation only, excluded from prompt
10-29: weak_reference, max 3, low-confidence label
30+: trusted_candidate
50+: operating_principle_candidate
```

Pattern expiry:

- unverified for 90 days: `SUSPECT`
- degraded after market regime change: downgrade candidate
- no automatic deletion
- no automatic promotion

## Storage And Recovery

Truth store:

```text
event_store: SQLite WAL + transaction + append-only lifecycle events
brain_store: FileLock + snapshot hash
daily_review: derived from event_store
```

Startup recovery:

```text
read ORDER_SENT / ORDER_ACKED / PARTIAL_FILLED / ORDER_UNKNOWN
re-query broker orders, fills, and balances
recover open positions from broker truth
keep ORDER_UNKNOWN when truth cannot be confirmed
```

## Phase Completion Gates

Every phase gate validates all previous phases again. This is required because later edits can contaminate earlier guarantees.

Example:

```powershell
python tools/v2_phase_gate.py --phase 1
python tools/v2_phase_gate.py --phase 2  # validates phase 1 and phase 2
python tools/v2_phase_gate.py --phase 6 --qa --simulation-report
```

Completion criteria:

- every `trade_ready` has `decision_id`
- event store persists across restart
- `SAFETY_PASSED` and `SAFETY_BLOCKED` are recordable
- `TIMING_EXPIRED` is recordable and remains forward-measure eligible
- daily review can be generated from event store
- tests pass

Gate:

```powershell
python tools/v2_phase_gate.py --phase 1
python -m unittest tests.test_v2_phase1
```

## Continuous Optimization Loop

Optimization is research-only by default. It must not mutate live config, live brain, or prompt rules automatically.

Command:

```powershell
python tools/v2_optimize_policy.py --min-trades 60 --min-validation-trades 20
```

Rules:

- use prior audit trade artifacts and daily reviews only
- no Claude calls
- split candidates into research-only and production-supported
- treat `same_close` and blank timing rows as research-only until a matching V2 Timing Adapter exists
- recommend only min-size live unless live CLEAN evidence has at least 20 trades, PF >= 1.15, average net PnL >= 0.10%, and no unresolved ORDER_UNKNOWN
- never auto-apply optimizer output to live config
- generate simulation report from previous simulation data and current V2 logic

The simulation may show weak or negative profitability. That is acceptable. The required output is a report that states how much return the current logic would have produced, and decomposes why it worked or failed without excessive Claude calls.

## Explicit Non-Goals

V2 does not add:

- new strategy
- ATR-based sizing
- uncontrolled Safety Gate additions
- Claude alpha re-judgment inside Safety Gate
- automatic brain promotion
- automatic paper-to-live brain merge
- ignored `ORDER_UNKNOWN`
- simultaneous full-system rewrites after weak returns
